# Route Interception & Auth Capture

This article covers how to capture authentication headers from browser traffic using Patchright's route interception. The intended audience is someone building browser-based integrations with APIs that use token authentication but don't offer OAuth or public API keys.

By the end, you will understand the interception pattern, how headers flow from browser to your application code, and how to manage the lifecycle of captured credentials.

---

## 1. The Problem

Some APIs authenticate exclusively through browser sessions. The user logs in through a web UI, the browser stores cookies and tokens, and all subsequent API calls include authorization headers generated by the application's JavaScript. There is no documented API, no OAuth flow, no way to get credentials except by being a logged-in browser.

To make programmatic API calls, you need to extract those headers from the browser's own traffic without disrupting the user's session.

---

## 2. Definitions

**Def. 1: Route Interception.** A mechanism where the automation framework intercepts HTTP requests made by the browser, allowing inspection or modification before the request reaches the network. In Patchright, this is `page.route(pattern, handler)`.

**Def. 2: Header Extraction.** The process of reading authentication headers from an intercepted request. The interceptor copies the headers it needs, then allows the original request to proceed unmodified.

**Def. 3: Session Lifecycle.** The state machine governing an authentication session: disconnected, connecting, needs-login, connected, error. Transitions are triggered by browser events and header capture.

**Def. 4: Token Refresh.** The process of detecting when captured credentials have expired and re-capturing fresh ones from new browser traffic. Bearer tokens typically expire after 24 hours; cookies after 30 days.

---

## 3. The Interception Pattern

The core idea is simple: attach a route handler to the browser page that matches API requests, copy the headers you need, then let the original request through.

```
Browser Page
    │
    ▼
[ page.route('**/api.example.com/**', handler) ]
    │
    ├──► Extract headers (Authorization, X-Custom-Token, ...)
    │         │
    │         ▼
    │    [ Store in memory ]
    │
    └──► route.fulfill() or route.continue()
              │
              ▼
         Original request proceeds to server
```

The browser doesn't know it's being intercepted. The API server receives exactly the request the browser intended to send. The only side effect is that your code now has a copy of the headers.

---

## 4. Basic Implementation

The simplest interceptor captures one header from any matching request:

```typescript
import type { Page, Route } from 'patchright';

let capturedToken: string | null = null;

await page.route('**/api.example.com/**', async (route) => {
  const request = route.request();
  const authHeader = request.headers()['authorization'];

  if (authHeader) {
    capturedToken = authHeader;
  }

  // Let the request proceed normally
  await route.continue();
});
```

Notice that `route.continue()` sends the original request to the server unchanged. The interception is read-only. The browser and the server both behave as if the interceptor doesn't exist.

**Note:** `route.continue()` is the correct method for pass-through interception. Using `route.fulfill()` would require you to make the HTTP request yourself and return the response, which adds complexity and can break streaming responses.

---

## 5. Structured Extraction with Validation

A production interceptor captures multiple headers and validates them. Without validation, you might store partial or malformed headers that fail when used.

We define the required headers as a Zod schema:

```typescript
import { z } from 'zod';

const HeadersSchema = z.object({
  Authorization: z.string().startsWith('Bearer '),
  'X-Api-Version': z.string(),
  'X-Timezone-Id': z.string(),
});

type CapturedHeaders = z.infer<typeof HeadersSchema>;
```

The interceptor extracts all required headers and validates them as a group:

```typescript
class ApiInterceptor {
  private headers: CapturedHeaders | null = null;
  private attached = false;

  async attach(page: Page): Promise<void> {
    if (this.attached) return;

    await page.route('**/api.example.com/**', async (route) => {
      const reqHeaders = route.request().headers();

      // Try to extract the headers we need
      const candidate = {
        Authorization: reqHeaders['authorization'] ?? '',
        'X-Api-Version': reqHeaders['x-api-version'] ?? '',
        'X-Timezone-Id': reqHeaders['x-timezone-id'] ?? '',
      };

      const result = HeadersSchema.safeParse(candidate);
      if (result.success) {
        this.headers = result.data;
      }

      await route.continue();
    });

    this.attached = true;
  }

  getHeaders(): CapturedHeaders | null {
    return this.headers;
  }
}
```

Notice that we parse with `safeParse`, not `parse`. If the headers are incomplete—which happens on non-API requests that match the URL pattern—we silently skip them. Only a fully valid header set gets stored.

**Note:** Request header names in Patchright are lowercased. The schema uses the original casing for readability, but extraction must use lowercase keys from `request.headers()`.

## 6. Waiting for Headers

The interceptor is passive—it only captures headers when the browser makes an API call. If you attach the interceptor and immediately check `getHeaders()`, it returns null because no API call has happened yet.

Add a polling method that waits for the first successful capture:

```typescript
async waitForHeaders(timeoutMs = 10000): Promise<CapturedHeaders | null> {
  const start = Date.now();

  while (Date.now() - start < timeoutMs) {
    if (this.headers) return this.headers;
    await new Promise(r => setTimeout(r, 200));
  }

  return null;
}
```

In practice, the first API call happens within a few seconds of page load. The wait is a safety net for slow connections, not a normal path.

## 7. Session Lifecycle

The authentication state follows a discriminated union:

```typescript
type AuthState =
  | { status: 'disconnected' }
  | { status: 'connecting' }
  | { status: 'needs-login'; sessionId: string }
  | { status: 'connected'; headers: CapturedHeaders; accountId: string }
  | { status: 'error'; message: string };
```

The lifecycle:

```
disconnected
    │
    ▼ startSession()
connecting
    │
    ├──► URL contains /login → needs-login
    │         │
    │         ▼ (user logs in manually)
    │     waitForHeaders()
    │         │
    │         ▼
    └──► headers captured → connected
              │
              ▼ token expires (24h)
         needs-refresh
              │
              ▼ new API call → headers re-captured
         connected (with fresh headers)
```

The key insight is that **login is manual**. The automation doesn't type passwords. It opens the login page, the user authenticates through the normal UI (including 2FA), and the interceptor passively captures the resulting auth headers from subsequent API calls.

This is both simpler and safer than automating login—you don't store passwords, and you don't fight CAPTCHA systems.

## 8. Persistent Profiles

Patchright's `launchPersistentContext` saves cookies and localStorage to disk. When you restart the browser with the same profile directory, the session resumes without re-login.

```typescript
const context = await chromium.launchPersistentContext(
  './data/browser-profiles/my-service',
  { headless: false }
);
```

Cookie-based sessions typically last 30 days. Bearer tokens last 24 hours. The profile preserves the cookies, so the browser stays logged in. When a Bearer token expires, the application's JavaScript transparently refreshes it, and the interceptor captures the new token from the refresh request.

**Note:** Profile directories contain authentication state. They must be in `.gitignore` and should not be shared between environments.

---

## 9. Detaching

When you're done intercepting, remove the route handler:

```typescript
async detach(): Promise<void> {
  if (!this.attached || !this.page) return;

  await this.page.unroute('**/api.example.com/**');
  this.attached = false;
}
```

Detaching is important for pages that stay open. Route handlers accumulate if you attach without detaching, and each handler runs on every matching request.

---

## 10. Using Captured Headers

Once you have the headers, you can make API calls from your server code without the browser:

```typescript
const headers = interceptor.getHeaders();
if (!headers) throw new Error('Not authenticated');

const response = await fetch('https://api.example.com/account', {
  headers: {
    Authorization: headers.Authorization,
    'X-Api-Version': headers['X-Api-Version'],
    'X-Timezone-Id': headers['X-Timezone-Id'],
    Accept: 'application/json',
  },
});
```

The API server sees a request with valid headers. It doesn't know or care that the headers came from interception rather than a browser.

**Note:** Captured headers are only valid for the session that produced them. If the user logs out in the browser, the headers become invalid immediately.

---

## 11. Summary

This article covered four concepts:

1. **Route interception** — `page.route()` with `route.continue()` for read-only header capture
2. **Structured extraction** — Zod validation ensures only complete, well-formed header sets are stored
3. **Session lifecycle** — a state machine from disconnected through manual login to connected
4. **Persistent profiles** — disk-backed browser profiles that survive restarts and preserve cookie sessions

The pattern generalizes to any service that authenticates through browser sessions. The interceptor is service-agnostic—only the URL pattern and the header schema change.

**Exercise:** Modify the interceptor to capture response headers in addition to request headers. Some services return session tokens in response headers (e.g., `X-Session-Token`) that must be included in subsequent requests.

**Further reading:** The session manager in the `@interceptor/browser` package implements disk-based credential persistence with automatic expiry detection.
